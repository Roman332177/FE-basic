Hometask

  • git config -global user.name blabla - дозволяє переглядати та настроювати параметри, що контролюють всі аспекти роботи Git, а також його зовнішній вигляд.
  • git config --global user.email blabal@bla
  • git init - створює новий репозиторій Git. З її допомогою можна перетворити існуючий проект без керування версіями на репозиторій Git або ініціалізувати новий порожній репозиторій.
  • git status показує стан файлів у робочому каталозі та індексі: які файли змінені, але не додані до індексу; які очікують коміти в індексі. На додаток до цього виводяться підказки про те, як змінити стан файлів.
  • git add- додає вміст робочого каталогу індекс (staging area) для наступного коммита. За промовчанням git commit використовує лише цей індекс, так що ви можете використовувати git add для складання зліпка наступного комміту.
  • git commit
  • git log - перегляд історії комітів
  • git checkout - мають на увазі перемикання між різними версіями цільового об'єкта. Команда git checkout працює з трьома різними об'єктами: файлами, коммітами та гілками. Під перемиканням зазвичай розуміють дію, пов'язане з виконанням команди git checkout.
  • git branch - Команду git branch переважно використовують для створення, перегляду, перейменування та видалення гілок.
  • git push - Команда git push використовується для вивантаження вмісту локального репозиторію у віддалений репозиторій. Вона дозволяє передати коміти з локального репозиторію у віддалений. Ця команда симетрична команді git fetch: при витягуванні за допомогою fetch коміти імпортуються в локальні гілки, а при публікації за допомогою push коміти експортуються у віддалені гілки.
  • git pull - Команда git pull використовується для вилучення та завантаження вмісту з віддаленого репозиторію та негайного оновлення локального репозиторію цим вмістом. Злиття видалених вищих змін у локальний репозиторій - це звичайне завдання робочого процесу, що виникає при спільній роботі на основі системи Git.
  • git merge - Злиття використовується в Git, щоб зібрати воєдино розгалужену історію. Команда git merge виконує злиття окремих напрямів розробки, створених за допомогою команди git branch у єдину гілку.
  • git clone - git clone - це утиліта командного рядка Git для вибору існуючого репозиторію та створення його клону, тобто копії.

Про файл .gitignore

  • .gitignore потрібен для приховання файлів та папок від системи контролю версій Git. Зазвичай приховують файли конфігурації (особливо з паролями), тимчасові файли і папки. gitignore використовує формат glob для вибірки файлів.
  • Основний синтаксис:
    • Кожен рядок – окремий шаблон
    • Порожні рядки ігноруються
    • Рядки, що починаються з # є коментарями
    • Символ сліша "/" на початку рядка вказує на поточну папку (де лежить .gitignore)
    • Зірочка(*) замінює будь-яку кількість символів
    • Дві зірочки (**) використовуються для вказівки всіх підпапок.
    • Знак оклику (!) на початку рядка інвертує шаблон (використовується для виключень)
    • Для екранування спецсимволів використовується зворотний слеш "\". Для ігнорування всієї директорії правило повинно закінчуватися на слеш(/), інакше правило вважається ім'ям файлу.

ADVANCED

  • git revert - використовується для відкату змін історії коммітів репозиторію. Інші команди скасування, такі як git checkout та git reset, переміщують вказівник HEAD та вказівники гілки на певний коміт. Команда git revert також працює з певним коммітом, проте використання git revert не переміщує покажчики. Під час операції revert відбувається перехід до зазначеного комміту, звертаються його зміни та створюється новий, «зворотний» коміт. Потім позиція покажчиків оновлюється - вони переміщуються до цього коміту наприкінці гілки.
  • git reset - це складний універсальний інструмент для скасування змін. Вона має три основні форми виклику, що відповідають аргументам командного рядка --soft, --mixed, --hard. Кожен із цих трьох аргументів відповідає трьом внутрішнім механізмам керування станом Git: дереву коммітів (HEAD), розділу проіндексованих файлів та робочому каталогу
  • git rebase - це процес переміщення послідовності коммітів до нового базового коміту чи їх об'єднання. Операцію перебазування найзручніше застосувати та відобразити у контексті створення функціональних гілок
  • git cherry-pick - бере зміни, що вносяться одним коммітом, і намагається повторно застосувати їх у вигляді нового комміту в поточній гілці. Ця можливість корисна в ситуації, коли потрібно забрати парочку коммітів з іншої гілки, а не зливати гілку цілком з усіма змінами, що внесені до неї.
  • git stash - зберігає непідтверджені зміни (індексовані та неіндексовані) в окремому сховищі, щоб ви могли повернутися до них пізніше. Потім відбувається відкат до початкової робочої копії.
